    문서 구성 (모두 포함):

    1. HTML 보일러플레이트 전체 예시
       - starter/html/index.html 기반 실제 코드 블록 포함
       - DOCTYPE, html lang="ko", meta charset="UTF-8", viewport, title, skip-nav, main 구조

    2. 필수 head 메타태그
       - meta charset="UTF-8" — 파일 인코딩과 동일하게 UTF-8
       - meta name="viewport" content="width=device-width, initial-scale=1" — 반응형 필수
       - meta http-equiv="X-UA-Compatible" content="IE=edge" — IE 호환성 (공공기관 요구시)
       - title 형식: 페이지 제목 | 사이트명

    3. lang 속성 규칙
       - html lang="ko" 필수 — 모든 HTML 파일에 적용
       - 영문 혼용 페이지: span lang="en" 으로 인라인 처리

    4. 본문건너뛰기 (skip navigation)
       - 항상 body 최상단에 위치
       - 코드 예시 포함 (Phase 05 패턴 — transition: none, 즉시 노출)
       - a href="#main-content" class="skip-nav" 로 구현
       - main id="main-content" 로 연결

    5. 시맨틱 태그 사용 기준
       - header: 사이트 전체 또는 섹션의 소개/내비게이션 영역
       - nav: 주요 네비게이션 링크 모음, aria-label로 구분
       - main: 페이지 주요 콘텐츠, 한 페이지에 하나만
       - section: 제목을 가진 독립적인 콘텐츠 구역
       - article: 독립적으로 배포 가능한 콘텐츠 (게시글, 카드)
       - aside: 보조 콘텐츠 (사이드바, 관련 링크)
       - footer: 저작권, 연락처, 사이트맵
       - Bad/Good 대조 예시 (div 남용 vs 시맨틱 태그)

    6. 기타 필수 규칙
       - type="button" 명시 — form 안 버튼은 반드시 type 명시
       - alt 속성 필수 — 장식용 alt="", 의미있는 이미지는 내용 설명
       - tabindex 양수 사용 금지 — 자연스러운 DOM 순서 유지
       - 이메일/전화번호 링크: a href="mailto:", a href="tel:"

       - 각 항목 설명: charset=utf-8, end_of_line=lf, indent_style=space, indent_size=2,

       - 이유: 에디터/OS 간 표시 차이 방지

       - 주석은 한국어로 작성 — 팀 내부 문서이므로
       - 섹션 구분 주석: // === 패턴
       - // TODO:, // FIXME: 접두사 규칙
       - 납품 전 제거 대상: // TEMP:, // DEBUG:

    4. 파일 인코딩 및 줄바꿈
       - 저장 시 UTF-8 (BOM 없음)
       - 줄바꿈: LF (CRLF 금지) — Git 설정과 연동
       - 파일 끝 빈 줄 1개 유지

    5. 에디터 설정 권장사항
       - VS Code 기준: EditorConfig 확장 설치
       - 저장 시 자동 포매팅 설정 (editor.formatOnSave: true)

       - type(scope): 한국어 설명 형식
       - 한국어 명령형 동사로 마무리: "추가", "수정", "제거", "개선"
       - 예시 3개 이상 포함

    2. 커밋 타입 분류표 (기존 index.md 이관 + 확장)
       - feat: 새 기능, 신규 컴포넌트 추가
       - fix: 버그 수정, 오류 교정
       - docs: 문서 변경 (코드 변경 없음)
       - style: 코드 포매팅, 세미콜론 등 (로직 변경 없음)
       - refactor: 코드 구조 개선 (기능 변경 없음)
       - chore: 빌드 설정, 패키지 관리 등
       - a11y: 접근성 개선 (팀 추가 타입)
       - perf: 성능 개선

    3. scope 예시
       - button, form, modal, tokens, docs, starter, build

    4. 좋은 커밋 / 나쁜 커밋 예시
       - Bad: fix: 수정함 / update / 작업중
       - Good: feat(button): .btn-ghost 커스텀 variant 추가 / fix(form): aria-invalid 누락 수정

    5. 브랜치 전략 (기존 index.md 이관)
       - main: 납품 가능 상태
       - develop: 개발 통합
       - feature/이슈번호-설명: 기능별 개발 브랜치
       - hotfix/이슈번호-설명: 긴급 수정 브랜치

    6. 커밋 원자성 원칙
       - 하나의 커밋 = 하나의 논리적 변경
       - 다수 파일 수정이라도 같은 목적이면 하나의 커밋
