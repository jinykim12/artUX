# KRDS v1.0.0 / KWCAG 2.1 AA 접근성 체크리스트

**기준 디자인시스템:** <a href="https://www.krds.go.kr" target="_blank" rel="noopener" title="새 창으로 열림">KRDS v1.0.0 (범정부 UI/UX 디자인시스템) <span class="sr-only">(새 창)</span></a>
**접근성 기준:** KWCAG 2.1 AA (24개 검사항목)
**우선순위:** 웹접근성 > 웹호환성 > 디자인 일관성

공공기관 납품 프로젝트 접근성 검수용 — 복사하여 프로젝트별 체크리스트로 사용

> **사용 방법:** 이 파일을 프로젝트 저장소에 복사한 뒤, 검수 완료 항목은 `- [ ]`를 `- [x]`로 변경하여 기록한다.
> 미달 항목은 해당 항목 아래에 이슈 내용과 조치 계획을 자유 형식으로 추가한다.

## 1. 인식의 용이성 (Perceivable)

<!-- 사용자가 정보를 인지할 수 있도록 콘텐츠를 표현 -->

- [ ] **1.1.1 적절한 대체 텍스트 제공** — 자동/수동: 수동
  - **왜 필요한가:** 스크린리더 사용자는 이미지를 볼 수 없으므로, alt 텍스트가 없으면 이미지의 내용을 전혀 파악할 수 없다. 공공기관 납품 시 KWCAG 1.1.1 필수 검사 항목이다.
  - 기준: 모든 의미 이미지에 목적을 설명하는 대체 텍스트(`alt`) 제공, 장식 이미지는 `alt=""`
  - 테스트: 이미지 비표시(CSS off) 후 alt 텍스트만으로 콘텐츠 이해 가능한지 확인

- [ ] **1.2.1 자막 제공** — 자동/수동: 수동
  - **왜 필요한가:** 청각 장애 사용자는 음성 정보를 들을 수 없다. 자막 없는 영상은 해당 사용자에게 완전히 차단된 콘텐츠이다.
  - 기준: 영상·음성 콘텐츠에 동기화된 자막 또는 대본 제공
  - 테스트: 영상 재생 시 자막 표시 여부, 음성만 있는 파일에 대본 링크 존재 확인

- [ ] **1.3.1 색에 무관한 콘텐츠 인식** — 자동/수동: 수동
  - **왜 필요한가:** 색각이상(색맹) 사용자는 빨강/초록 같은 특정 색 조합을 구분하지 못한다. 색상만으로 오류를 표시하면 이 사용자들은 오류가 발생했는지조차 알 수 없다.
  - 기준: 색상 만으로 정보를 전달하지 않음 (예: 오류 상태를 빨간색+아이콘+텍스트로 표시)
  - 테스트: 색맹 시뮬레이터(NoCoffee, Coblis) 또는 흑백 화면으로 정보 인식 가능 여부 확인

- [ ] **1.3.2 명확한 지시 사항 제공** — 자동/수동: 수동
  - **왜 필요한가:** 스크린리더 사용자는 화면의 "위치", "색상", "모양"을 인식하지 못한다. "오른쪽 빨간 버튼을 클릭하세요" 같은 안내는 그들에게 무의미하다.
  - 기준: 위치·색·모양·크기만으로 조작 대상을 지정하지 않음 (예: "오른쪽 빨간 버튼 클릭" ❌)
  - 테스트: 지시 문구가 시각 정보 없이도 이해되는지 검토

- [ ] **1.3.3 텍스트 콘텐츠의 명도 대비** — 자동/수동: 자동+수동
  - **왜 필요한가:** 저시력 사용자, 밝은 햇빛 아래 사용자, 노인 등은 대비가 낮은 글자를 읽기 어렵다. 이 항목 미달 시 KWCAG 1.3.3 위반으로 공공기관 납품 심사에서 탈락 사유가 된다.
  - 기준: 일반 텍스트 4.5:1 이상, 18pt(24px) 이상 또는 볼드 14pt(19px) 이상 대형 텍스트 3:1 이상
  - 테스트: Chrome DevTools → Accessibility → Contrast Ratio, 또는 axe DevTools 자동 검사

- [ ] **1.3.4 자동 재생 금지** — 자동/수동: 수동
  - **왜 필요한가:** 자동 재생 영상/음성은 스크린리더 음성과 충돌하여 사용자가 화면 내용을 전혀 들을 수 없게 한다. 광과민성 발작 위험도 있다.
  - 기준: 3초 이상 자동 재생 콘텐츠(영상·음성·애니메이션)는 정지/중지 수단 제공
  - 테스트: 페이지 로드 후 자동 재생 여부, 정지 버튼 키보드 접근 가능 여부 확인

- [ ] **1.4.1 콘텐츠 간의 구분** — 자동/수동: 수동
  - **왜 필요한가:** 색상만으로 콘텐츠 블록을 구분하면 색각이상 사용자나 흑백 화면 사용자가 섹션 경계를 인식하지 못한다.
  - 기준: 인접한 콘텐츠 블록이 시각적으로 구분 가능 (여백, 선, 배경색 등으로)
  - 테스트: 콘텐츠 블록 경계가 색상 없이(또는 색맹 상태에서) 인식 가능한지 확인

## 2. 운용의 용이성 (Operable)

<!-- 사용자가 인터페이스 컴포넌트를 조작할 수 있도록 -->

- [ ] **2.1.1 키보드 사용 보장** — 자동/수동: 수동
  - **왜 필요한가:** 지체 장애나 운동 장애 사용자는 마우스를 사용할 수 없다. 키보드로 접근이 불가능한 기능은 이들에게 사용 불가 서비스가 된다. KWCAG 2.1.1 위반 시 공공기관 납품 불가이다.
  - 기준: 마우스 없이 키보드(Tab, Enter, Space, 방향키)만으로 모든 기능 사용 가능
  - 테스트: 마우스 없이 Tab 키만으로 전체 기능 탐색 및 실행 가능 여부 확인

- [ ] **2.1.2 초점 이동** — 자동/수동: 수동
  - **왜 필요한가:** 포커스가 모달이나 특정 영역에 갇히거나 예측 불가능하게 이동하면 키보드 사용자가 페이지를 탐색하다 길을 잃는다.
  - 기준: 키보드 초점이 특정 요소에 갇히지 않으며 순서에 맞게 이동
  - 테스트: Tab 순서가 시각적 레이아웃 순서와 일치하고 모달 등에서 포커스 트랩이 의도적으로만 동작하는지 확인

- [ ] **2.1.3 조작 가능** — 자동/수동: 수동
  - **왜 필요한가:** 드래그·슬라이드 등 복잡한 제스처는 운동 장애 사용자나 보조 기기 사용자가 수행하기 어렵다. 반드시 키보드나 단순 클릭으로 동일한 기능을 이용할 수 있어야 한다.
  - 기준: 드래그·슬라이드·멀티터치 등 제스처 인터랙션에 키보드/단일 포인터 대안 제공
  - 테스트: 슬라이더, 드래그앤드롭 등 복합 조작 UI에 키보드 대안 존재 여부 확인

- [ ] **2.2.1 응답시간 조절** — 자동/수동: 수동
  - **왜 필요한가:** 읽기 속도나 입력 속도가 느린 사용자는 타임아웃 전에 작업을 완료하지 못할 수 있다. 세션이 갑자기 만료되면 입력한 내용을 전부 잃는다.
  - 기준: 세션 만료·타임아웃이 있을 경우 연장·해제·경고 수단 제공
  - 테스트: 자동 로그아웃 경고 메시지 및 연장 버튼 존재, 키보드 접근 가능 여부 확인

- [ ] **2.2.2 정지 기능 제공** — 자동/수동: 수동
  - **왜 필요한가:** 자동으로 움직이는 배너나 티커는 스크린리더 사용자가 해당 내용을 읽기 전에 넘어가 버린다. 또한 주의력 장애 사용자에게 방해 요소가 된다.
  - 기준: 자동 갱신·스크롤 콘텐츠(배너, 티커 등)에 정지·일시정지 기능 제공
  - 테스트: 자동 슬라이드, 자동 스크롤 배너에 정지 버튼 존재 및 키보드 접근 가능 여부 확인

- [ ] **2.3.1 깜빡임과 번쩍임 사용 제한** — 자동/수동: 수동
  - **왜 필요한가:** 초당 3회 이상 번쩍이는 요소는 광과민성 발작(간질)을 유발할 수 있다. 생명과 직결되는 안전 항목이다.
  - 기준: 1초에 3회 이상 깜빡이거나 번쩍이는 콘텐츠 금지
  - 테스트: 애니메이션·영상에서 고주파 플래시 요소 없는지 육안·도구(PEAT) 확인

- [ ] **2.4.1 반복 영역 건너뛰기** — 자동/수동: 수동
  - **왜 필요한가:** 스크린리더나 키보드 사용자가 매 페이지마다 GNB(메인 메뉴) 전체를 처음부터 탐색해야 한다면 매우 불편하다. "본문 바로가기" 링크 하나로 이 문제를 해결한다.
  - 기준: 반복 GNB·헤더 등 건너뛰기 링크(`skip to content`) 최상단에 제공
  - 테스트: Tab 1회 후 "본문 바로가기" 링크 포커스 여부, Enter 클릭 시 본문으로 이동 여부 확인

- [ ] **2.4.2 제목 제공** — 자동/수동: 자동+수동
  - **왜 필요한가:** 스크린리더 사용자는 페이지 제목과 heading 구조로 페이지의 전체 윤곽을 파악한다. 제목이 없거나 무의미하면("새 페이지", "제목 없음") 현재 어느 페이지에 있는지 알 수 없다.
  - 기준: 페이지·섹션·프레임에 설명적 제목(`<title>`, `<h1>`~`<h6>`) 제공
  - 테스트: axe DevTools 자동 검사 + 스크린리더로 페이지 제목 및 heading 구조 확인

- [ ] **2.4.3 적절한 링크 텍스트** — 자동/수동: 자동+수동
  - **왜 필요한가:** 스크린리더는 "링크 목록 보기" 기능으로 페이지 안의 링크를 문맥 없이 나열한다. "여기를 클릭", "더 보기" 같은 텍스트가 10개 나열되면 각 링크가 어디로 연결되는지 전혀 알 수 없다.
  - 기준: 링크 텍스트만으로 목적지·기능 이해 가능 (예: "여기를 클릭" ❌, "예약 신청서 다운로드" ✓)
  - 테스트: axe DevTools로 빈 링크·이미지 링크 자동 검사 + 링크 목록으로 문맥 없이 이해 가능한지 확인

## 3. 이해의 용이성 (Understandable)

<!-- 정보와 인터페이스 작동 방식을 사용자가 이해할 수 있도록 -->

- [ ] **3.1.1 기본 언어 표시** — 자동/수동: 자동
  - **왜 필요한가:** 스크린리더는 `<html lang>` 속성을 기반으로 올바른 언어 엔진으로 텍스트를 읽는다. `lang="ko"` 없이 영문 엔진으로 한국어를 읽으면 발음이 완전히 깨져 내용을 알아들을 수 없게 된다.
  - 기준: `<html lang="ko">` 등 페이지 기본 언어 명시
  - 테스트: axe DevTools 자동 검사, 또는 HTML 소스에서 `<html lang>` 속성 확인

- [ ] **3.2.1 사용자 요구에 의한 실행** — 자동/수동: 수동
  - **왜 필요한가:** select 값을 바꾸었을 뿐인데 페이지가 자동으로 전환되면 스크린리더 사용자는 갑자기 다른 페이지에 놓여 현재 위치를 잃는다. 사용자의 명시적 "제출" 행동 없이는 페이지가 이동해서는 안 된다.
  - 기준: 포커스 이동·값 변경만으로 페이지 전환·팝업이 자동 실행되지 않음
  - 테스트: select, radio 등에서 변경만으로 예기치 않은 페이지 이동·팝업이 발생하지 않는지 확인

- [ ] **3.3.1 콘텐츠의 선형 구조** — 자동/수동: 수동
  - **왜 필요한가:** 스크린리더는 DOM 순서대로 콘텐츠를 읽는다. CSS로 시각 순서를 바꿔도 DOM 순서가 바뀌지 않으면 읽는 순서와 보이는 순서가 달라져 내용이 뒤죽박죽이 된다.
  - 기준: DOM 순서와 시각 순서가 일치하여 CSS 없이 읽어도 의미가 통함
  - 테스트: CSS 비활성 상태에서 콘텐츠 순서가 논리적으로 유지되는지 확인

- [ ] **3.4.1 레이블 제공** — 자동/수동: 자동+수동
  - **왜 필요한가:** 스크린리더 사용자가 입력 필드에 포커스를 올렸을 때 레이블이 없으면 "텍스트 편집" 이라고만 들린다. 어떤 정보를 입력해야 하는지 알 수 없다. 공공기관 납품 심사에서 가장 자주 적발되는 항목 중 하나이다.
  - 기준: 모든 입력 요소에 연결된 `<label>` 또는 `aria-label`, `aria-labelledby` 제공
  - 테스트: axe DevTools 자동 검사 + 스크린리더로 폼 입력 시 레이블 읽힘 여부 확인

- [ ] **3.4.2 오류 정정** — 자동/수동: 수동
  - **왜 필요한가:** 스크린리더 사용자는 오류가 발생해도 빨간 테두리나 아이콘 변화를 인식하지 못한다. 오류 내용이 텍스트로 제공되지 않으면 무엇이 잘못되었는지, 어디를 수정해야 하는지 알 수 없다.
  - 기준: 입력 오류 시 오류 위치·내용 텍스트로 안내, 수정 방법 제안
  - 테스트: 폼 제출 오류 시 오류 메시지가 텍스트로 표시되고 해당 필드로 포커스가 이동하는지 확인

## 4. 견고성 (Robust)

<!-- 다양한 보조 기술에서 콘텐츠를 해석할 수 있도록 -->

- [ ] **4.1.1 마크업 오류 방지** — 자동/수동: 자동
  - **왜 필요한가:** 중복 ID나 잘못된 태그 구조는 스크린리더가 DOM을 올바르게 파싱하지 못하게 만들어 예측 불가능한 동작을 유발한다. 자동 검사로 쉽게 잡히므로 반드시 통과시켜야 한다.
  - 기준: 중복 ID 없음, 시작/종료 태그 올바름, 속성 중복 없음
  - 테스트: W3C HTML Validator (validator.w3.org), axe DevTools 자동 검사

- [ ] **4.2.1 웹 애플리케이션 접근성 준수** — 자동/수동: 자동+수동
  - **왜 필요한가:** JavaScript로 만든 동적 UI(모달, 탭, 드롭다운 등)는 HTML 기본 기능이 없으므로 WAI-ARIA 없이는 스크린리더가 해당 요소의 역할과 상태를 전혀 파악하지 못한다.
  - 기준: 자바스크립트 기반 동적 UI에 WAI-ARIA 역할·상태·속성 올바르게 사용
  - 테스트: axe DevTools 자동 검사 + 스크린리더(NVDA/VoiceOver)로 동적 컴포넌트 동작 확인

- [ ] **4.2.2 웹 콘텐츠 기술 규격 준수** — 자동/수동: 자동
  - **왜 필요한가:** 비표준 마크업은 브라우저와 보조 기술마다 다르게 해석되어 특정 환경에서 접근성이 완전히 깨질 수 있다.
  - 기준: 사용된 마크업 언어 공식 규격 준수 (HTML5, ARIA 명세)
  - 테스트: W3C HTML Validator + axe DevTools ARIA 규칙 자동 검사

## 자동 검사 도구

> **참고:** 자동 검사는 전체 접근성 오류의 약 30~40%만 탐지 가능하다. 반드시 수동 검사를 병행한다.

| 도구 | 용도 | 방법 |
|------|------|------|
| **Lighthouse** | 접근성 점수 + 자동 항목 일괄 검사 | Chrome DevTools → Lighthouse → Accessibility 카테고리 실행 |
| **axe DevTools** | 상세 WCAG 위반 항목 + 코드 위치 제시 | Chrome 확장 설치 후 DevTools → axe 탭에서 Analyze 실행 |
| **pa11y** | CLI 기반 자동 검사 (CI/CD 통합 예정 — v2) | `npx pa11y https://example.com --standard WCAG2AA` |
| **W3C Validator** | HTML 마크업 오류 검사 | validator.w3.org 또는 `npx vnu --format json [file.html]` |

## 수동 검사 방법

### 키보드 탐색 (필수)

1. 마우스를 사용하지 않고 **Tab** 키로 페이지 전체 이동
2. **Shift+Tab**: 역방향 이동
3. **Enter / Space**: 버튼·링크 활성화
4. **방향키**: 드롭다운·탭·라디오버튼 내 이동
5. **Escape**: 모달·드롭다운 닫기
6. 모든 대화형 요소에 포커스 표시(outline)가 명확히 표시되는지 확인

### 스크린리더 (권장)

| 환경 | 도구 | 단축키 참고 |
|------|------|------------|
| Windows | **NVDA** (무료) + Chrome | NVDA 공식 사용자 가이드 |
| macOS / iOS | **VoiceOver** (내장) | Cmd+F5(Mac), 세 손가락 탭(iOS) |
| Android | **TalkBack** (내장) | 접근성 → TalkBack 활성화 |

**스크린리더 검사 체크포인트:**
- 페이지 타이틀이 콘텐츠를 설명하는가
- heading 구조(h1→h2→h3)가 논리적인가
- 폼 레이블이 올바르게 읽히는가
- 이미지 alt 텍스트가 의미 있게 읽히는가
- 동적 변경사항(알림, 오류)이 즉시 알림되는가

## KRDS v1.0.0 컴포넌트 대응표

이 프로젝트에서 구현한 컴포넌트가 KRDS v1.0.0 기준 어떤 컴포넌트에 대응하는지 정리한다.
KRDS 프레임워크를 직접 사용하지는 않지만, 접근성 패턴과 ARIA 속성은 KRDS 명세를 따른다.

| artpqUX 컴포넌트 | KRDS v1.0.0 대응 | KRDS 카테고리 | 핵심 접근성 속성 |
|----------------|-------------------|---------------|------------------|
| 헤더 / GNB | Header + Main menu | 아이덴티티 + 탐색 | `aria-label`, `aria-expanded`, `aria-controls` |
| 본문건너뛰기 | Skip link | 탐색 | `href="#main-content"` |
| 브레드크럼 | Breadcrumb | 탐색 | `aria-label="breadcrumb"`, `aria-current="page"` |
| 페이지네이션 | Pagination | 탐색 | `aria-label`, `aria-current="page"` |
| 버튼 | Button | 액션 | `type` 명시, `aria-label` (아이콘 버튼) |
| 폼 (입력) | Text input, Textarea, Select | 입력 + 선택 | `<label>`, `aria-required`, `aria-invalid`, `aria-describedby` |
| 모달 | Modal | 레이아웃 및 표현 | `role="dialog"`, `aria-modal`, `aria-labelledby`, 포커스 트랩 |
| 탭 | Tab | 레이아웃 및 표현 | `role="tablist/tab/tabpanel"`, `aria-selected`, `aria-controls` |
| 테이블 | Table | 레이아웃 및 표현 | `<caption>`, `scope`, `<thead>/<tbody>` |
| 카드 | — (KRDS 미정의) | — | 시맨틱 마크업, 링크 내 이미지 `alt` |
| 슬라이더 | Carousel | 레이아웃 및 표현 | `aria-live`, 정지 버튼, `aria-label` |

### KRDS에서 참조할 추가 패턴

artpqUX에 아직 없지만 KRDS v1.0.0에서 공공기관 납품 시 자주 요구되는 패턴:

| KRDS 컴포넌트 | 카테고리 | 핵심 접근성 요건 |
|---------------|----------|------------------|
| Masthead (공식 배너) | 아이덴티티 | 정부 사이트 식별 배너, 최상단 고정 |
| Identifier (운영기관 식별자) | 아이덴티티 | 운영 기관명 + 로고, 푸터 내 배치 |
| Accordion (아코디언) | 레이아웃 및 표현 | `aria-expanded`, `aria-controls`, `hidden` |
| Disclosure (디스클로저) | 레이아웃 및 표현 | `aria-expanded`, FAQ/약관 펼침 |
| Toast / Snackbar | 피드백 (모바일) | `role="alert"`, `aria-live` |
| Step indicator (단계 표시기) | 피드백 | `aria-current="step"`, 진행 상태 안내 |
